A módosításokon sorozatán átesett több verziójú **adatok** hatékony kezelését **verziókezelésnek** nevezzük.
A verziókezelést a gyakorlatban **VCS** (Version Control System) néven illetjük.
## A verziókövetés

A verziókövető rendszerek lényege, hogy egy projekt során az egymás mellett párhuzamosan futó verziókat tudják kezelni. Ennek a célja **a forráskód változásainak követése, illetve a korábbi állapotok kezelése**.
## Alapfogalmak

1. **Repository** (tároló) – a projekthez tartozó összes fájlt tartalmazó „adatbázis”. Elhelyezkedés szerint a következő típusai vannak:
    2. **local** (helyi) – a repository a saját helyi gépünkön van
    3. **remote** (távoli) – a repository egy távoli szerveren van
    4. Hozzáférés alapú csoportosítása a repóknak:  
        5. **public** – nyilvánosan mindenki eléri  
        6. **private** – csak engedélyezett személyek férnek hozzá
2. **Revision** – a forrásállományok egy **változata** (verzió)
3. **Working copy / working directory** (munkakönyvtár) – az adott projekthez tartozó fájlokat tartalmazza a helyi számítógépen
4. **Snapshot** (pillanatkép) – a fájlok és mappák egy adott pillanatban vett aktuális tartalma
5. **Staged / cached fájl** – módosított állomány, amelyről már készült snapshot
6. **Commit** – a snapshotok létrehozása, biztonságos tárolása az „adatbázisban”. Ha az utolsó commit óta nem történik változás a working directory-ban, akkor a directory **tiszta**.
7. **Branch** – a fejlesztés egy adott commit eredményéből származó, saját névvel ellátott **ága**.
8. **Checkout** – a különböző branchek közötti váltás
9. **Merge** – a branchek egyesítése és a párhuzamos fejlesztések összeolvasztása
10. **Conflict** – akkor fordul elő, ha ugyanazt a programsort különbözőképpen módosították a fejlesztők. Ezt megbeszélés alapján kódösszefésüléssel oldják meg.
11. **Head** – egy adott branch-ben a legutolsó commitra való hivatkozás
12. **Base** – a working copy-ban található legfrissebb változat
13. **Tag** – az egyes revisionökhöz rendelhető egyedi címkék (pl.: 2.3)
14. **Trunk / baseline / master** – a fejlesztés fő branch-e.

## A verziókezelés modellek

1. **CVCS** (Centralized Version Control System) – **központosított verziókezelés**, során a projekt összes verzióját egy központi szerver tárolja  
    2. SVN  
    3. CVS Subversion
2. **DVCS** (Distributed Version Control System) – az **elosztott verziókezelő rendszerek** a projekt verzióit a szerveren és a fejlesztők gépein is tárolják.  
    5. Mercurial  
    6. Git

# Git

Nyílt forráskódú DVCS.
Előnyei:
1. A saját, elszigetelt környezetünkben tudunk dolgozni vele.
2. Rendkívül gyors a verziókezelési műveletek sora.
A rendszer sajátosságai:
3. Van a directory és a local repo között egy köztes állapota, amit **staging area** (adatfeldolgozó terület) nevezünk.
4. Egy rejtett `.git` mappa jön létre egy új repo létrehozásakor.
5. A Git rendszer alapértelmezett branch neve a master / main.
## Git parancsok

1. `git add <fájlnév>` – A fájlokat a directoryból a staging area-ba helyezi.
2. `git commit -m "üzenet"` – A staging area-ból a fájlokat a local repóba helyezi, és üzenettel látja el őket.
3. `git checkout` – A local repóban lévő fájlokat lehet vele szerkesztésre visszavonni, ez rögtön a working directory-ba helyezi őket.
4. `git push` – A remote repóba tölti fel a local repo tartalmát.
5. `git pull` – A remote repo tartalmának letöltése a local repóba.

## Fájlstátuszok Gitben

1. **Untracked** – A Git nem követi a változásokat, és nem készít snapshotot a fájlról.
2. **Tracked** – A Git követi a fájl változásait, és három új állapotot különböztet meg:  
    3. **unmodified** – a fájl nem módosult az utolsó commit óta  
    4. **modified** – a fájl módosult az utolsó commit óta  
    5. **staged / cached** – a fájl módosult, és készült róla snapshot

## Mi az a GitHub?

1. Használható remote repo gyanánt.
2. Rajta keresztül **egyszerűen adminisztrálhatók** a fejlesztési problémák, issue-k:  
    3. Lehetőséget nyújt új, megoldatlan issue-k létrehozására  
    4. Az issue-k fejlesztőkhöz rendelhetők  
    5. Az issue-k megoldására új brancheket is létre lehet hozni.

## Git parancsok

1. `git config --global user.name "név"` – A Git felhasználónév beállítása
2. `git config --global user.email "email"` – A Git felhasználó email beállítása
3. `git init` – Üres repo létrehozása
4. `git status` – Információ a könyvtárról és a benne lévő fájlokról
5. `git add fájlnév` – Egy vagy több fájl hozzáadása. Az összes fájl: `git add .`
6. `git commit -m "üzenet"` – A pillanatkép tárolóba helyezése
7. `git log` – Napló lekérése
8. `git branch név` – Branch létrehozása
9. `git checkout branch_név` – Váltás az adott branchre
10. `git merge branch_név` – A másik branch egyesítése az aktuálissal
11. `git clone repo_elérés` – A repository klónozása
12. `git push repo_elérés` – Commitok feltöltése
13. `git pull repo_elérés` – Változások letöltése és egyesítése
